## 0. 为什么值得做 / 为什么后置

- **杀手锏能力轴**:对家庭陪伴产品,"小七,把客厅灯关了""太热了开下空调"是体感最强的一类指令——比影音/天气更"像个管家"。
- **后置的原因**:依赖用户家里**真有**智能设备 + **有可对接的入口**;接入工程不小(见下);且控制家电有现实后果(误操作的代价高于文件/影音)。故先预研、定接缝,不进当前一期。

## 1. 国内现实(决定接入路线)

- **米家(小米 IoT)** 普及率最高,但**官方开放平台面向企业/商用**,个人开发者拿授权难;社区方案 `python-miio` 走**局域网 token**,但 token 提取繁琐、固件更新易失效、设备覆盖不全 —— 不稳,不适合"开箱即用"产品。
- **Home Assistant(HA)** 开源、**本地部署(局域网)**、有标准 **REST API + WebSocket**,且已**聚合**米家 / 涂鸦 / 海尔 / Aqara 等几十个品牌(社区集成)。对接 HA 一个 API ≈ 覆盖多品牌。
- **逐品牌直连**(米家 + 涂鸦 + 海尔 + 小度…):碎片化、每家一套鉴权与协议、维护爆炸 —— **否**(同 robot 插件病的教训)。

## 2. 路线结论:HA 作一期接入目标

- **本地局域网 → 天然"国内通畅"**(无墙、无境外依赖,合天气块定下的硬约束),延迟低。
- **HA 当聚合层**:Larkwing 只实现"对接 HA"一条,品牌差异由 HA 吸收 —— 与宪法 §4 **"多供应商同构、源=数据"**、§5 **trait 接缝** 完全一致。这是该哲学的**第 4 个实例**:
  | 能力域 | trait | 一期实现 |
  |---|---|---|
  | LLM | `LlmProvider` | DeepSeek(openai_compat) |
  | 影音 | `MediaSource` | bilibili |
  | 天气 | `WeatherSource` | Open-Meteo / 和风 |
  | **智能家居** | **`HomeControl`** | **HomeAssistant** |
- 米家直连**不做一期**(授权 + token 不稳);留作 `HomeControl` 的另一实现,真有用户强需求且能拿到稳定入口时再加(trait 逃生口已在)。

## 3. `HomeControl` trait 草图(同 MediaSource/WeatherSource)

```rust
// core/src/home/mod.rs(届时)
pub struct Device {
    pub id: String,          // HA 的 entity_id,如 light.living_room
    pub name: String,        // 友好名(HA 里用户起的)
    pub area: Option<String>,// 区域:客厅/卧室(HA 提供)
    pub kind: DeviceKind,    // Light | Switch | Climate | Sensor | Cover | ...
    pub state: String,       // on/off/温度值…(原样透传,渲染交给上层)
}

#[async_trait]
pub trait HomeControl: Send + Sync {
    async fn list_devices(&self) -> Result<Vec<Device>>;
    async fn set_state(&self, id: &str, action: Action) -> Result<()>; // 开关/调亮度/调温…
}
// 实现:HaSource { base_url, token, http }  —— REST 读状态 + POST 调用服务;
// 实时状态推送(灯被别人关了)可走 HA 的 WebSocket,二期。
```

- **app 级无归属资产**(HTTP client + token):住工具单例字段,**不进 ToolCtx**(同 `WeatherClient`/`WebClient`,§4 三物种判据)。

## 4. 工具形状预想(正交原语,不造任务工具)

- **`home_devices`(读)**:列设备 + 当前状态,可按区域/类型过滤。模型先看清家里有什么、什么状态。
- **`home_control`(写)**:对某设备执行动作(开/关、调亮度、设温度…)。**批量原生**(一句"睡觉了"关多个灯 → 一次多设备),token 不随设备数爆(同 fs 写原语纪律)。
- **不造** `turn_on_living_room_light` 这类 per-device 任务工具 —— 通用原语 + `home_devices` 给的设备清单,模型自行组合(宪法 §5 "任务知识零入")。
- 设备名/区域是**环境知识**:可进 `briefing`(家庭备忘)让模型常驻知道"主卧灯叫什么",或每次 `home_devices` 现取(频率决定常驻与否,同宪法 §7 前缀准入判据)。

## 5. 风险分级与"执行前确认"(home_control 是头号候选)

- `home_control` = **`ToolRisk::Mutating`**(开关家里电器有现实后果,代价高于文件误操作)。
- `Tool::risk()` 的预留 slot(见 `tools/mod.rs`)**至今无人消费**;智能家居是**未来"执行前确认闸门"的首批、最强候选**——误关老人房间的氧气机比误删文件更需要一道闸。
- 一期立场:沿用 fs 的"不强制确认"(收敛、不烦人),但**记档为闸门首位**;真做控制时重新评估是否对 `Mutating` 家电动作加一句口头确认(尤其安防/门锁/燃气类高危设备,可能需要**设备级**风险标注,不是一刀切)。

## 6. 配置接入(同天气 key 哲学,§4 钥匙是用户的)

- HA 地址(`http://homeassistant.local:8123` 之类)+ **长效访问令牌**(HA 用户profile 生成)进 `settings`(app 级):
  - token 是**秘密** → 复用天气和风 key 的**掩码桥**纪律(`set_setting` 收 + `list_settings` 只回 `····尾4`,不明文过桥;见 `engine::ProviderView` / weather key 先例)。
  - 地址非秘密 → 走 `APP_SETTING_KEYS` 明文往返。
- **设置·高级新增"智能家居"段**(同"天气服务"段):地址输入 + token 输入 + HA 官网/文档链接(`window.open` 跳转,§ 引导式上手)+ "已连接 N 个设备"状态行。
- 没配 = 工具在场但调用即返回"还没接智能家居,去设置接一下"(友好退路,铁律 §3.5);**绝不**让没有设备的家庭看见一堆控制 UI。

## 7. 主动性联动(记档,不一期)

- 条件提醒已具备(PLAN 天气块):未来可扩到**设备动作**——"太热了自动开空调""下雨自动关窗"。
- 但**自动控制家电的安全边界远高于"提醒"**(自动开关电器无人在场风险大)。预研记档:自动控制需要更强的用户显式授权 + 可见可撤销,**不随条件提醒顺手做**。

## 8. 人格中立(宪法 §5)

- core 全链路只发**设备 key/状态/动作结果**数据;"灯开好啦~"这类话由当前人格组织(同影音 `AuthRequired`、天气数据的先例)。前端字典出 `tool.home_devices`/`tool.home_control` 的友好动词("瞅瞅家里的灯…""动一下开关…")。

## 9. 边界 / 不做(宪法 §9 延伸)

- 不自己做**设备发现协议**(mDNS/各家私有协议)—— 交给 HA。
- 不做**本地语音直控低延迟优化**(先走正常回合循环;延迟敏感再议)。
- 不做**米家直连**(除非强需求 + 稳定入口)。
- 不复刻 robot 的可配置化;**强默认**:接 HA 一条路走顺,别给普通人一堆协议选项。

## 10. 路线分期建议(真做时搬进 PLAN,每期独立可玩/独立绿)

1. **A. `HomeControl` trait + `HaSource`(读)**:`home/` 模块 + 对接 HA REST 拉设备清单与状态;`home_devices` 工具;设置接入(地址 + token 掩码桥)。验收:能列出家里的灯/开关及状态。
2. **B. `home_control`(写)**:开关 / 调亮度 / 调温;批量;`ToolRisk::Mutating`;few-shot(关灯、调暖)。验收:口头"关客厅灯"真关。
3. **C. 设置·高级"智能家居"段** + 友好退路 + "已连接 N 个设备"状态 + HA 文档链接。
4. **D.(后置)实时状态**:HA WebSocket 推送(别人关了灯,7274 也知道);**条件联动自动控制**(高授权门槛,单独设计)。

## 11. 真机 watch-items(届时验证)
